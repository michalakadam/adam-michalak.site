Code reviews are crucial for skills develepment of both author and reviewer. Critical but fair feedback is one of the best antidotes for confirmation bias. It hurts and seems harsh but believe me this pain is the call to growth. On the other hand investigating code written but others forces a developer to clarify their views on programming. Reading code is also a great way to learn about alternative approaches to problem solving. This article aims at explaining how to use git and GitHub to make your code easy to review, hence increasing likelihood of receiving detailed and insightful feedback about your code.
Holy master

Imagine that you are developing a product that is used by thousands of users. Many of them are requesting a new feature and you decide to provide it to them. You start coding and commit frequently. Soon frustrated users start reaching out to you saying that the program stopped working correctly. What happened? You committed changes to a production branch (master) and pushed them to the remote repo. An anditode to this problem might be to develop feature locally on your machine and push only when it is fully functional. However, imagine that users started reporting a serious bug in your product when the feature you are working on is half done... Should you get rid of all the code you have written so far and start working on a bug instead?

Fortunately git enables splitting project into easy-to-manage branches. How to use this feature effectively? First step is to make a commitment to keep master branch pristine.Treat master as a holy branch, reserved for fully-functional, production-ready and reviewed code.

Let's see how to apply this strategy from the very moment project begins. Create a new folder, initialize git repository with git init and add all the boilerplate files (like .gitignore, pom.xml, etc.) manually or use a script to do it for you. Check if everything is working correctly by compiling and executing whatever code you have. If there is no error thrown, it is the right time for git commit -m "initial commit". Now before you write a single line of code, switch to a new "development" branch with git checkout -b development. This is a place for a half-broken and/or not fully-developed code. Add commits to it until your app is working again. Push both master and development branches to a remote repository on GitHub. It is the right time to ask for a code review!

Pull requests 101

Pull request is a GitHub feature that makes code reviews efficent. It presents changes committed to a given branch in a user-friendly way. Thanks to that reviewer can directly point to a place in the code where correction is needed. Pull request provides tools for initiating a meaningful discussion that can lead to implementation of better solutions. Last but not least it supports holy state of master branch by not allowing merger until reviewer accepts all changes that are under scrutiny.

Let's see how to create a pull request from scratch. If you followed steps from the previous paragraph, open repository page on GitHub. First of all grant a permission for a reviewer to see code you created. To do that go to the repository settings page, select "Manage access" section and invite a reviewer of your choice. Then navigate to "Code" tab of the repository and click on the "New pull request" button that is located right above the list of files. Select master as a base branch and development as a branch that will be compared. After clicking "Create pull request" button you will be redirected to a pull request page. Locate "Reviewers" section on the right hand side and assign a reviewer. If you encounter any problem along the way, check this visual guide for a more detailed instruction.

Applying suggested changes

After review is completed you might see a screen like this:

At this point you might wonder how to address changes that reviewer requested. It is not possible to change commits that are pushed to remote repository and reverting remote changes may result in an unexpected behavior and wipe out reviewer's comments. Just create a new commit addressing changes requested by the reviewer and push it to the reviewed branch. They will appear in the pull request and the reviewer will be able to take a look at your code once more. If on the other hand you don't agree with the review, first of all make sure that you are not taking feedback too personally. Then form your response in the most polite way you can think of. Foster a discussion by presenting your point clearly. Remember that saying to anybody that they are plain wrong will only antagonize them and make coming up with the best solution impossible. It is also the easiest way to make sure that a given reviewer will never take a look at your code again.

Putting it all together

If you want to get your code reviewed, use holy master branching strategy combined with GitHub's pull request feature. Develop new code on a separate branch and merge it only when reviewer accepts it. If reviewer requests changes, address them with a new commit. If you want to learn more, check official github documentation on pull requests or read about a successful branching model.
